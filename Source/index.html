Play is quite a powerful concept to add to one's potfolio. If you are familiar with Java, you will love Play Framework. The documentation is easy to approach and understand. With already-done projects provided to guide users through the different stages. One can hardly get tired with the ideas therein.

Play Framework is an open-source web application framework which follows the model–view–controller (MVC) architectural pattern. It is written in Scala and usable from other programming languages that are compiled to JVM Bytecode, e.g. Java. It aims to optimize developer productivity by using convention over configuration, hot code reloading and display of errors in the browser. Play was created by software developer Guillaume Bort - Wikipedia

The goal of this page is to demonstrate Play Framework using Scala and ScalaJS. There are other pages out there, by other authors, that are also sufficient to do more. They are recommended. To start, go there to downlaod a starter project; or you may go to their Github page ; or you may proceed to clone directly the whole package from my repository . Project Structure

One can easily notice three most important folders in the project structure above. The first is client folder, which holds the ScalaJS files. Exploring server folder

This folder holds our Play application just like mentioned above. Notice that within this folder, there is controller, model, and view folders; the standard Play structure. It is within this controller class that you decide what the user should and should not see. For instance, you may decide the template to serve the user from this controller page. You can still write as many Scala codes and methods as you desire to meet your demand or the demand of your application within this controller class. You can also pass values from the controller class to your individual templates and use them within the template.

Note: the template engine used in Play is Twirl. It's okay to have a little understanding of Twirl to be able to navigate your way across your template files just fine.

Twirl is quite easy to use and understand. Template engines allow us to do more in HTML. HTML on its own is just a markup language; but with template engines, we can write for loops, while loops, conditional statements, and so on within our HTML script. The model folder.

This folder is very important in PlayFramework. It, along with the other two -- controllers and views -- complete the Play paradigm. The createTables.scala script needs to be properly configured in order to run it successfully. The Tables script seen included within this folder is generated automatically each time createTables.scala script is run successfully. It is not a good idea to change the contents of the Tables script. Although if you break this file, you can still come to this createTables.scala script and run it again to auto-generate a new one.

The routes file

This file is another important file in Play Framework. You can not navigate across multiple pages without the routes file. Please see below:

The routes script is located in the conf folder. Within this script, you define the different paths relevant to your application. You can see from above the different paths defined, which are important to this particular project. This script must be contacted first before the controller. For instance: in your JavaScript page, or even template page, to be able to navigate to another page, you have to specify the path first.

Looking at line 158 of the above picture, you see that secretary which is on line 19 of the routes script is specified as the path to follow here. So what will happen here is that our program will go first to the routes script to look for this particular path. Because it is there, it will proceed with the controller.UsersController.secretary() and invoke that secretary method in the controller class. Within this method, our template is served to the user.
